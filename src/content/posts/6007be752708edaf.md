# GBFR 存档修改工具：因子编辑功能与存档解析原理

## 前言

《碧蓝幻想：Relink》（Granblue Fantasy: Relink，简称 GBFR）是一款动作 RPG 游戏。游戏中"因子"（Sigil）是角色的核心装备系统，每个因子携带 1~2 个"特性"（Trait），决定了角色的属性加成和技能效果。

在游戏后期，玩家需要大量刷取特定组合的因子，这个过程非常耗时。基于对游戏存档格式的逆向分析，我开发了一个存档修改工具，支持离线生成、查看、删除因子。

本文介绍工具的因子编辑功能实现，以及 GBFR 存档文件的二进制解析原理。

## 功能概览

工具基于 Wails v2（Go + Vue 3）构建，包含三部分功能：

| 功能 | 实现方式 | 说明 |
|------|----------|------|
| PE 补丁 | 直接修改 exe 二进制 | 修改任务挑战次数上限、点赞数值 |
| 角色使用次数 | 运行时内存读写 | 通过 Windows API 读写游戏进程内存 |
| 因子生成 | 离线存档解析写入 | 解析 `.dat` 存档的 FlatBuffer 结构，读写因子数据 |

本文重点介绍第三部分——因子编辑与存档解析。

## 因子编辑功能

### 功能设计

因子编辑功能包括：

- **查看已有因子**：加载存档后自动展示所有因子，显示中文名称、等级、主/副特性
- **生成因子**：搜索选择因子，配置等级和特性，写入输出存档
- **批量操作**：支持将多个不同配置加入队列，按数量批量生成
- **删除因子**：支持勾选多个因子批量删除，或一键清除全部因子
- **中文本地化**：所有因子名、特性名均已翻译为简体中文

### 数据来源

因子数据来自三个嵌入式 JSON 文件：

1. **sigils.json**：因子定义（ID、32 位哈希、名称、允许等级、主/副特性列表）
2. **traits.json**：特性定义（ID、哈希、名称、最大等级、出现规则）
3. **secondary-trait-rules.json**：副特性兼容性规则（互斥关系、禁用列表）

此外还维护了一个从 Cheat Engine Table 提取的 `ctHashToName` 映射表（861 条），用于在 JSON 数据缺失时通过哈希值直接反查中文名称。

### 校验流程

用户添加因子到队列时，工具进行完整校验：

1. 因子 ID 必须在数据目录中存在
2. 因子等级必须在允许的等级列表中
3. 主特性必须存在，且等级合法
4. 如果因子支持副特性（V+ 因子），副特性必须在兼容列表中，且等级合法
5. 不支持副特性的因子，不可携带副特性

## 存档解析原理

这是本文的核心。GBFR 的存档文件（`.dat`）使用了一种自定义的二进制格式，结合了 FlatBuffer 风格的数据结构和 XXHash64 校验机制。

### 文件整体结构

```
[文件头 - 0x34+ 字节]
  offset 0x00-0x1B: 未知字段
  offset 0x1C-0x23: slot_data_offset  (uint64) — 槽位数据区相对于文件头的偏移
  offset 0x24-0x2B: 未知字段
  offset 0x2C-0x33: slot_data_length  (uint64) — 槽位数据区的长度

[槽位数据区]
  大量的变长 FlatBuffer 条目（unit entries），flat 排列

[哈希表]
  位于 slot_data_offset + slot_data_length - 0x14 处
  包含 10 个 XXHash64 哈希值，保护槽位数据区的 10 个分段
```

关键洞察：整个存档的核心数据都集中在"槽位数据区"中，只要找到这个区域并理解其中条目的结构，就能读写任何游戏数据。

### FlatBuffer 条目结构

每个条目代表一个键值对，结构如下：

```
[Table Header]  (位于条目开头)
  int32 vtable_dist   — 指向 VTable 的相对偏移

[VTable]  (位于 table_addr + vtable_dist)
  uint16 vtable_size       — VTable 自身大小
  uint16 object_size       — 对象总大小
  uint16 id_field_offset   — IDType 字段在 table 中的偏移
  uint16 unitid_field_offset — UnitID 字段的偏移 (可为 0)
  uint16 data_field_offset — ValueData 向量的偏移

[Fields]
  uint32  IDType    — 数据类型标识
  uint32  UnitID    — 实例/槽位标识 (可选)
  vector  ValueData — 一个或多个 uint32/int32 值
```

这里 IDType 和 UnitID 的组合构成了一个二维索引系统：

- **IDType** 表示"这是什么类型的数据"（如：因子 ID、特性哈希、等级等）
- **UnitID** 表示"这个数据属于哪个实例"（如：哪个因子的第几个特性槽）

### 关键 IDType 常量

| 常量 | 值 | 含义 |
|------|-----|------|
| HashSeedIDType | 1003 | 校验分段种子 |
| TraitHashIDType | 1701 | 特性哈希值 |
| TraitLevelIDType | 1702 | 特性等级 |
| GemSlotIDType | 2701 | 因子槽位编号 |
| GemIDType | 2703 | 因子 ID（哈希） |
| GemLevelIDType | 2704 | 因子等级 |
| GemWornByIDType | 2706 | 装备角色哈希 |
| GemFlagsIDType | 2707 | 标志位（普通因子为 2） |

### 因子槽位的数据模型

一个因子占 1 个槽位（slot_index），包含以下 FlatBuffer 条目：

```
Slot Entry:    IDType=2701, UnitID=GEM_SLOT_BASE_ID + slot_index
Gem Entry:     IDType=2703, UnitID=GEM_SLOT_BASE_ID + slot_index
Level Entry:   IDType=2704, UnitID=GEM_SLOT_BASE_ID + slot_index
WornBy Entry:  IDType=2706, UnitID=GEM_SLOT_BASE_ID + slot_index
Flags Entry:   IDType=2707, UnitID=GEM_SLOT_BASE_ID + slot_index
TraitHash:     IDType=1701, UnitID=TRAIT_SLOT_BASE + slot_index * 100 + 0
TraitLevel:    IDType=1702, UnitID=TRAIT_SLOT_BASE + slot_index * 100 + 0
TraitHash:     IDType=1701, UnitID=TRAIT_SLOT_BASE + slot_index * 100 + 1  (副特性)
TraitLevel:    IDType=1702, UnitID=TRAIT_SLOT_BASE + slot_index * 100 + 1  (副特性)
```

其中：
- `GEM_SLOT_BASE_ID = 30000`
- `TRAIT_SLOT_BASE = 120000000`
- 空槽位的标记值为 `0x887AE0B0`（`EmptyHash`）

### 条目定位算法

由于存档中的条目是 flat 排列的，没有索引结构，需要扫描定位。工具使用两种策略：

**策略一：字节扫描**

遍历槽位数据区的每个字节，寻找匹配的目标 IDType 值（4 字节）。找到后，向前最多搜索 20 字节来定位 Table Header（通过 VTable 校验确认有效性）。

这一策略利用了 FlatBuffer 的特征：VTable 的 object_size 必须大于 data_field_offset，且所有字段偏移必须在 object_size 范围内。

**策略二：4 字节对齐回退**

对于策略一可能遗漏的条目（某些条目的 IDType 可能被其他数据"包裹"），进行第二遍扫描：以 4 字节对齐方式遍历所有位置，逐位置尝试解析。

### 原地修改策略

工具采用**原地修改**的方式写入数据，而非反序列化-修改-序列化：

1. 将整个存档文件读入 `[]byte`
2. 扫描定位每个需要修改的 FlatBuffer 条目
3. 直接在 `[]byte` 中覆盖对应偏移的 uint32 值
4. 修改完成后写入新文件

这样做的好处是：
- 不需要完整理解所有 FlatBuffer 字段
- 不会破坏未知字段的数据
- 实现简洁，不易引入序列化错误

### 校验和修复

修改数据后，必须修复校验和，否则游戏会判定存档损坏。

GBFR 使用 **XXHash64** 算法。槽位数据区的末尾有一个哈希表，包含 10 个 uint64 校验值，分别保护数据区的 10 个等分区间。

修复流程：
1. 从槽位数据中读取 HashSeed（IDType=1003 的条目）
2. `section_index = hash_seed % 10`
3. 计算对应分段的 XXHash64
4. 覆盖哈希表中对应位置的 uint64 值

关键代码示意：

```go
const seed = 0x2F1A43EBCD
sectionStart := slotDataStart + slotDataLen/10*sectionIndex
sectionEnd   := slotDataStart + slotDataLen/10*(sectionIndex+1)
hash := xxhash.Sum64S(slotData[sectionStart:sectionEnd], seed)
binary.LittleEndian.PutUint64(slotData[hashTableStart+sectionIndex*8:], hash)
```

### 验证机制

写入后，工具会重新加载输出文件并逐字段验证：
- 重新定位每个写入的条目
- 读取实际值并与期望值比对
- 任一字段不匹配即报告验证失败

## 逆向过程回顾

存档格式的逆向分析过程如下：

1. **对比差分**：在游戏中做一次操作（如装备因子），保存前后两份存档，用十六进制编辑器对比差异
2. **定位数据结构**：通过差异定位到因子的哈希值、等级、特性等字段在存档中的位置
3. **发现 IDType 规律**：观察到数据使用 `(IDType, UnitID, Value)` 三元组组织，识别出 FlatBuffer 结构
4. **校验和逆向**：修改数据后游戏提示存档损坏，通过 code cave 注入定位到 XXHash64 调用，提取 seed 和分段逻辑
5. **交叉验证**：对比多份存档、参考社区项目（如 GBFR-Sigil-Generator）验证解析正确性

## 结语

这个工具的存档解析方案虽然看起来"暴力"——扫描整个数据区来定位条目——但这种方式恰好避免了过度依赖游戏版本的问题。只要 FlatBuffer 的 VTable 结构和 IDType 常量不变，工具就能正常工作。

因子生成功能的加入使得玩家可以快速获得想要的因子组合，大幅减少了重复刷取的时间。
