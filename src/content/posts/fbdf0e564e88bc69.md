# mongodb分片集群和安全

### <font style="color:rgb(0, 0, 0);">2.1 分片概念</font>
<font style="color:rgb(34, 34, 34);">分片 (</font>**<font style="color:rgb(34, 34, 34);">sharding</font>**<font style="color:rgb(34, 34, 34);">) 是一种跨多台机器分布数据的方法， MongoDB 使用分片来支持具有非常大的数据集和高吞吐量操作的部署。</font>

<font style="color:rgb(34, 34, 34);">换句话说：分片 (sharding) 是指将数据拆分，将其分散存在不同的机器上的过程。有时也用分区 (</font>**<font style="color:rgb(34, 34, 34);">partitioning</font>**<font style="color:rgb(34, 34, 34);">) 来表示这个概念。将数据分散到不同的机器上，不需要功能强大的大型计算机就可以储存更多的数据，处理更多的负载。</font>

<font style="color:rgb(34, 34, 34);">具有大型数据集或高吞吐量应用程序的数据库系统可以会挑战单个服务器的容量。例如，高查询率会耗尽服务器的 CPU 容量。工作集大小大于系统的 RAM 会强调磁盘驱动器的</font><font style="color:rgb(34, 34, 34);"> </font><font style="color:rgb(34, 34, 34);">I/O</font><font style="color:rgb(34, 34, 34);"> </font><font style="color:rgb(34, 34, 34);">容量。</font>

<font style="color:rgb(34, 34, 34);">有两种解决系统增长的方法：</font>**<font style="color:rgb(34, 34, 34);">垂直扩展</font>**<font style="color:rgb(34, 34, 34);">和</font>**<font style="color:rgb(34, 34, 34);">水平扩展</font>**<font style="color:rgb(34, 34, 34);">。</font>

+ **<font style="color:rgb(34, 34, 34);">垂直扩展</font>**<font style="color:rgb(34, 34, 34);">意味着</font>**<font style="color:rgb(34, 34, 34);">增加单个服务器的容量</font>**<font style="color:rgb(34, 34, 34);">，例如使用更强大的CPU，添加更多RAM或增加存储空间量。可 用技术的局限性可能会限制单个机器对于给定工作负载而言足够强大。此外基于云的提供商基于可用的硬件配置具有硬性上限。结果，垂直缩放有实际的最大值。</font>
+ **<font style="color:rgb(34, 34, 34);">水平扩展</font>**<font style="color:rgb(34, 34, 34);">意味着</font>**<font style="color:rgb(34, 34, 34);">划分系统数据集并加载多个服务器</font>**<font style="color:rgb(34, 34, 34);">，添加其他服务器以根据需要增加容量。虽然单个机器的总体速度或容量可能不高，但每台机器处理整个工作负载的子集，可能提供比单个高速大容量服务器更高的效率。扩展部署容量只需要根据需要添加额外的服务器，这可能比单个机器的高端硬件的总体 成本更低。权衡是基础架构和部署维护的复杂性增加。</font>

<font style="color:rgb(34, 34, 34);">MongoDB 支持通过</font>**<font style="color:rgb(34, 34, 34);">分片进行水平扩展</font>**<font style="color:rgb(34, 34, 34);">。</font>

### <font style="color:rgb(0, 0, 0);">2.2 分片集群包含的组件</font>
<font style="color:rgb(34, 34, 34);">MongoDB 分片群集包含以下组件：</font>

+ <font style="color:rgb(34, 34, 34);">分片（存储）：每个分片包含分片数据的子集。 每个分片都可以部署为副本集。</font>
+ <font style="color:rgb(34, 34, 34);">mongos （路由）：mongos充当查询路由器，在客户端应用程序和分片集群之间提供接口。</font>
+ <font style="color:rgb(34, 34, 34);">conﬁg servers （”调度” 的配置）：配置服务器存储群集的元数据和配置设置。 从MongoDB 3.4 开始，必须将配置服务器部署为副本集（CSRS）。</font>

### <font style="color:rgb(0, 0, 0);">2.3 分片集群架构目标</font>
<font style="color:rgb(136, 136, 136);">两个分片节点副本集（3+3）+ 一个配置节点副本集（3）+ 两个路由节点（2），共 11 个服务节点</font>

## 2.4配置服务副本集
1.在根目录创建log和data\db文件夹

2.在根目录创建mongod.conf配置文件

3.在配置中配置分片角色分片角色（sharding：clusterRole：）为configsvr

只有两个参数（shardsvr 或  configsvr）副本集分片和配置分片 可选

```java
# 数据保存到哪
storage:
  dbPath: D:\Developer\MongoDB666\mongodb-shard-27021\data
  journal:
    enabled: true
#  engine:
#  mmapv1:
#  wiredTiger:
# 日志保存到哪
systemLog:
  destination: file
  logAppend: true
  path:  D:\Developer\MongoDB666\mongodb-shard-27021\log\mongod.log
# 绑定的IP和端口号
net:
  port: 27021
  bindIp: 127.0.0.1
# 复制集名称
replication:
  replSetName: 'myconfigrs'
# 复制集的作用：是一个配置服务器
sharding:
		#分片角色
    clusterRole: configsvr
```

## 2.4分片节点的搭建
与常规副本集搭建一直，不同的是须在配置文件中加配置信息（分片角色sharding：clusterRole：）

只有两个参数（shardsvr 或  configsvr）副本集分片和配置分片 可选

```java
# 数据保存到哪
storage:
  dbPath: D:\Developer\MongoDB666\mongodb-shard-27021\data
  journal:
    enabled: true
#  engine:
#  mmapv1:
#  wiredTiger:
# 日志保存到哪
systemLog:
  destination: file
  logAppend: true
  path:  D:\Developer\MongoDB666\mongodb-shard-27021\log\mongod.log
# 绑定的IP和端口号
net:
  port: 27021
  bindIp: 127.0.0.1
# 复制集名称

  replSetName: 'itzb'

		#分片角色

```

## 2.5路由节点的搭建
1.在路由节点根目录中创建log文件夹

2.在根目录创建mongos.conf配置文件，指定配置节点副本集

```java

systemLog:
  destination: file
  logAppend: true
  path: /opt/mongodb/logs/router.log

net:
  port: 27000
  bindIp: 0.0.0.0

processManagement:
  timeZoneInfo: /usr/share/zoneinfo
  fork: true

security:
  keyFile: /opt/mongodb/sharding-sync-key.file

sharding:
  # 配置中心地址，你有几台就配置几台
  configDB: myconfigrs/10.20.1.156:27010,10.20.1.157:27010,10.20.1.158:27010,10.20.1.159:27010
```

进行完成以上操作后路由节点和配置节点建立管理但无法存储数据，因为和分片节点未建立联系，需要添加分片节点

<font style="color:rgb(34, 34, 34);">添加分片</font>

```java
sh.addShard("IP:Port", "IP:Port", "IP:Port") 
```

<font style="color:rgb(34, 34, 34);">查看分片状态情况</font>

```java
sh.status() 
```

<font style="color:rgb(34, 34, 34);">如果添加分片失败，需要先手动移除分片，检查添加分片的信息的正确性后，再次添加分片。 </font>

<font style="color:rgb(34, 34, 34);">移除分片:</font>

```java
use admin $ db.runCommand( { removeShard: "myshardrs02" } ) 
```

+ <font style="color:rgb(34, 34, 34);">如果只剩下最后一个 shard，是无法删除的</font>
+ <font style="color:rgb(34, 34, 34);">移除时会自动转移分片数据，需要一个时间过程</font>
+ <font style="color:rgb(34, 34, 34);">完成后，再次执行删除分片命令才能真正删除</font>
+ <font style="color:rgb(34, 34, 34);">开启分片功能</font>

```java
sh.enableSharding("articledb") 
    
sh.enableSharding("库名")
    
sh.shardCollection("库名.集合名",{"key":1}) 
```

<font style="color:rgb(34, 34, 34);">集合分片，使用</font><font style="color:rgb(34, 34, 34);"> </font><font style="color:rgb(34, 34, 34);">sh.shardCollection()</font><font style="color:rgb(34, 34, 34);"> </font><font style="color:rgb(34, 34, 34);">方法指定集合和分片键</font>

```java
sh.shardCollection(namespace, key, unique) 
```

<font style="color:rgb(34, 34, 34);">对集合进行分片时, 你需要选择一个</font><font style="color:rgb(34, 34, 34);"> </font>**<font style="color:rgb(34, 34, 34);">片键</font>**<font style="color:rgb(34, 34, 34);"> </font><font style="color:rgb(34, 34, 34);">(Shard Key) shard key 是每条记录都必须包含的, 且建立了</font>**<font style="color:rgb(34, 34, 34);">索引</font>**<font style="color:rgb(34, 34, 34);">的单个字段或复合字段, MongoDB按照片键将数据划分到不同的数据块中,并将数据块均衡地分布到所有分片中. 为了按照片键划分数据块, MongoDB使用</font>**<font style="color:rgb(34, 34, 34);">基于哈希</font>**<font style="color:rgb(34, 34, 34);">的分片方式（随机平均分配）或者</font>**<font style="color:rgb(34, 34, 34);">基于范围</font>**<font style="color:rgb(34, 34, 34);">的分片方式（数值大小分配） 。</font>

<font style="color:rgb(136, 136, 136);">用什么字段当片键都可以，如：nickname作为片键，但一定是必填字段。</font>

### <font style="color:rgb(0, 0, 0);">分片策略(规则)</font>
#### <font style="color:rgb(34, 34, 34);">哈希策略</font>
<font style="color:rgb(34, 34, 34);">对于</font><font style="color:rgb(34, 34, 34);"> </font>**<font style="color:rgb(34, 34, 34);">基于哈希</font>**<font style="color:rgb(34, 34, 34);"> </font><font style="color:rgb(34, 34, 34);">的分片 , MongoDB计算一个字段的哈希值, 并用这个哈希值来创建数据块.</font>

<font style="color:rgb(34, 34, 34);">在使用基于哈希分片的系统中, 拥有”相近”片键的文档</font>**<font style="color:rgb(34, 34, 34);">很可能不会</font>**<font style="color:rgb(34, 34, 34);">存储在同一个数据块中, 因此</font>**<font style="color:rgb(34, 34, 34);">数据的分离性</font>**<font style="color:rgb(34, 34, 34);">更好一些.</font>

#### <font style="color:rgb(34, 34, 34);">范围策略</font>
<font style="color:rgb(34, 34, 34);">对于</font><font style="color:rgb(34, 34, 34);"> </font>**<font style="color:rgb(34, 34, 34);">基于范围</font>**<font style="color:rgb(34, 34, 34);"> </font><font style="color:rgb(34, 34, 34);">的分片 , MongoDB 按照片键的范围把数据分成不同部分. 假设有一个数字的片键 : 想象一个从负无穷到正无穷的直线,每一个片键的值都在直线上画了一个点. MongoDB把这条直线划分为更短的不重叠的片段, 并称之为数据块 ,每个数据块包含了片键在一定范围内的数据.</font>

<font style="color:rgb(34, 34, 34);">在使用片键做范围划分的系统中, 拥有”相近”片键的文档</font>**<font style="color:rgb(34, 34, 34);">很可能存储在同一个数据块</font>**<font style="color:rgb(34, 34, 34);">中, 因此也会存储在同一个分片中.</font>

#### <font style="color:rgb(34, 34, 34);">基于范围的分片方式与基于哈希的分片方式性能对比</font>
<font style="color:rgb(34, 34, 34);">基于范围的分片方式提供了更高效的范围查询, 给定一个片键的范围,分发路由可以很简单地确定哪个数 据块存储了请求需要的数据,并将请求转发到相应的分片中. 不过, 基于范围的分片会导致数据在不同分片上的不均衡,有时候,带来的消极作用会大于查询性能的积极作用. 比如, 如果片键所在的字段是线性增长的, 一定时间内的所有请求都会落到某个固定的数据块中, 最终导致分布在同一个分片中. 在这种情况下, 一小部分分片承载了集群大部分的数据,系统并不能很好地进行 扩展. 与此相比, 基于哈希的分片方式以范围查询性能的损失为代价, 保证了集群中数据的均衡.哈希值的随机性 使数据随机分布在每个数据块中, 因此也随机分布在不同分片中.但是也正由于随机性, 一个范围查询很难 确定应该请求哪些分片, 通常为了返回需要的结果,需要请求所有分片.</font>

<font style="color:rgb(34, 34, 34);">如无特殊情况，一般推荐使用 Hash Sharding. 而使用 _id 作为片键是一个不错的选择，因为它是必有的，你可以使用数据文档 _id 的哈希作为片键。 这个方案能够是的读和写都能够平均分布，并且它能够保证每个文档都有不同的片键所以数据块能够很 精细。 似乎还是不够完美，因为这样的话对多个文档的查询必将命中所有的分片。虽说如此，这也是一种比较 好的方案了。 理想化的 shard key 可以让 documents 均匀地在集群中分布</font>

## <font style="color:rgb(0, 0, 0);">3. 安全认证</font>
### <font style="color:rgb(0, 0, 0);">3.1 MongoDB的用户和角色权限简介</font>
<font style="color:rgb(34, 34, 34);">默认情况下，MongoDB实例启动运行时是没有启用用户访问权限控制的，也就是说，在实例本机服务 器上都可以随意连接到实例进行各种操作，MongoDB不会对连接客户端进行用户验证，这是非常危险 的。</font>

<font style="color:rgb(34, 34, 34);">mongodb官网上说，为了能保障mongodb的安全可以做以下几个步骤</font>

1. <font style="color:rgb(34, 34, 34);">使用新的端口，默认的 27017 端口如果一旦知道了 ip 就能连接上，不太安全</font>
2. <font style="color:rgb(34, 34, 34);">设置 mongodb 的网络环境，最好将 mongodb 部署到公司服务器内网，这样外网是访问不到的。公 司内部访问使用 vpn 等</font>
3. <font style="color:rgb(34, 34, 34);">开启安全认证。认证要同时设置服务器之间的内部认证方式，同时要设置客户端连接到集群的账号 密码认证方式。</font>

<font style="color:rgb(34, 34, 34);">为了强制开启用户访问控制(用户验证)，则需要在MongoDB实例启动时使用选项 –auth 或在指定启动 配置文件中添加选项</font><font style="color:rgb(34, 34, 34);"> </font><font style="color:rgb(34, 34, 34);">auth=true</font>

<font style="color:rgb(34, 34, 34);">在开始之前需要了解一下概念</font>

#### <font style="color:rgb(34, 34, 34);">启用访问控制</font>
+ <font style="color:rgb(34, 34, 34);">MongoDB使用的是基于角色的访问控制(Role-Based Access Control,RBAC)来管理用户对实例的访问。 通过对用户授予一个或多个角色来控制用户访问数据库资源的权限和数据库操作的权限，在对用户分配 角色之前，用户无法访问实例</font>
+ <font style="color:rgb(34, 34, 34);">在实例启动时添加选项 –auth 或指定启动配置文件中添加选项</font><font style="color:rgb(34, 34, 34);"> </font><font style="color:rgb(34, 34, 34);">auth=true</font>

#### <font style="color:rgb(34, 34, 34);">角色</font>
<font style="color:rgb(34, 34, 34);">在MongoDB中通过角色对用户授予相应数据库资源的操作权限，每个角色当中的权限可以显式指定， 也可以通过继承其他角色的权限，或者两都都存在的权限。</font>

#### <font style="color:rgb(34, 34, 34);">权限</font>
<font style="color:rgb(34, 34, 34);">权限由指定的数据库资源(resource)以及允许在指定资源上进行的操作(action)组成</font>

1. <font style="color:rgb(34, 34, 34);">资源(resource)包括：数据库、集合、部分集合和集群</font>
2. <font style="color:rgb(34, 34, 34);">操作(action)包括：对资源进行的增、删、改、查(CRUD)操作</font>

<font style="color:rgb(34, 34, 34);">在角色定义时可以包含一个或多个已存在的角色，新创建的角色会继承包含的角色所有的权限。在同一 个数据库中，新创建角色可以继承其他角色的权限，在 admin 数据库中创建的角色可以继承在其它任意 数据库中角色的权限。</font>

```java
# 查询所有角色权限(仅用户自定义角色)
$ db.runCommand({ rolesInfo: 1 })

# 查询所有角色权限(包含内置角色)
$ db.runCommand({ rolesInfo: 1, showBuiltinRoles: true })


# 查询当前数据库中的某角色的权限
$ db.runCommand({ rolesInfo: "<rolename>" })

# 查询其它数据库中指定的角色权限
$ db.runCommand({ rolesInfo: { role: "<rolename>", db: "<database>" } }

# 查询多个角色权限
$ db.runCommand({
  rolesInfo: [
    "<rolename>",
    {
      role: "<rolename>",
      db: "<database>"
    },
    ...

  ]
})

```

<font style="color:rgb(34, 34, 34);">常用的内置角色：</font>

+ <font style="color:rgb(34, 34, 34);">数据库用户角色：read、readWrite</font>
+ <font style="color:rgb(34, 34, 34);">所有数据库用户角色：readAnyDatabase、readWriteAnyDatabase、 userAdminAnyDatabase、dbAdminAnyDatabase</font>
+ <font style="color:rgb(34, 34, 34);">数据库管理角色： dbAdmin、dbOwner、userAdmin</font>
+ <font style="color:rgb(34, 34, 34);">集群管理角色： clusterAdmin、clusterManager、clusterMonitor、hostManager</font>
+ <font style="color:rgb(34, 34, 34);">备份恢复角色： backup、restore</font>
+ <font style="color:rgb(34, 34, 34);">超级用户角色： root</font>
+ <font style="color:rgb(34, 34, 34);">内部角色： system</font>
